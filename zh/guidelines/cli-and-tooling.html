CLI 和工具指南

在运行 CLI 命令、排查命令错误或为任务选择合适的工具时，请使用此指南。它是 FaasJS 工具链的快速参考，旨在减少命令执行错误。

默认工作流

  1. 在运行任何 FaasJS 命令之前，先用 npm install 安装依赖。
  2. 创建、重命名或移动 .api.ts 文件后，运行 faas types 重新生成类型声明。
  3. 提交前使用 vp check --fix 进行代码格式化和 lint 检查。
  4. 使用 vp test 作为默认测试运行器；使用 vp test <pattern> 进行聚焦运行。
  5. 当有待处理迁移时，通过 faasjs-pg migrate 运行数据库迁移。
  6. 当命令不可用时，回退到 npx <command> 并检查 node_modules 是否存在。

faas CLI

由 @faasjs/dev 提供，作为 faas 二进制（faas.mjs）。它有两个子命令：run 和 types。

命令 描述
faas run <file> 使用预加载的 FaasJS Node 模块钩子运行 TypeScript 文件
faas run <file> --root <path> 指定项目根目录以解析 <file>（默认：process.cwd()）
faas types 生成 API 类型声明到 src/.faasjs/types.d.ts
faas types --root <path> 指定类型生成的项目根目录

全局选项：

  • -h, --help — 显示帮助文本。
  • -v, --version — 打印 @faasjs/dev 版本。

运行详情：faas run 解析 @faasjs/node-utils/register-hooks 并生成一个带有 --import 的子 Node 进程，使目标脚本获得正常的 process.argv。

类型生成详情：faas types 扫描 src/ 中的 .api.ts 文件，将文件名转换为路由，并写入 src/.faasjs/types.d.ts。它会跳过 .faasjs 和 node_modules 目录。只有 .api.ts 文件和 faas.yaml 的变更会触发类型重新生成（参见 @faasjs/dev 中的 isTypegenInputFile）。

Vite Plus / vp

由 vite-plus 包提供。它是基于 Vite 和 Vitest 的开发与构建工具层。

命令 描述
vp check --fix 通过 oxlint 和 oxfmt 运行代码检查（lint）和格式化修复
vp test 用 Vitest 运行所有测试
vp test <pattern> 运行匹配文件名校验的测试
vp test --watch 以监听模式运行测试
vp dev 启动开发服务器

常用组合：

# 交接前运行检查和测试
vp check --fix && vp test

# 运行特定测试文件
vp test src/pages/users/api/__tests__/list.test.ts

# 以监听模式运行匹配校验的测试
vp test list --watch

# 使用特定日志级别运行测试
FaasLog=info vp test

# 在自定义端口上启动开发服务器（在 vite.config.ts 中配置）
vp dev

项目创建

使用 create-faas-app 脚手架创建新项目。

npx create-faas-app --name <project-name>

选项：

  • --name <name> — 项目文件夹名称。
  • --template <template> — 模板名称（默认：admin）。

可用模板：

  • admin（默认）— 推荐的 React + Ant Design + PostgreSQL 启动模板。
  • minimal — 更轻量的 React 启动模板。

脚手架完成后，脚本会自动运行：

  1. npm install — 安装依赖。
  2. npm run test — 运行初始测试套件以验证设置。

要在创建后手动运行这些步骤：

cd <project-name>
npm install
npx faas types

迁移命令

由 @faasjs/pg 作为 faasjs-pg 二进制提供。

命令 描述
faasjs-pg new <name> 在 migrations/ 中创建新的带时间戳的迁移文件
faasjs-pg status 显示所有迁移的状态
faasjs-pg migrate 运行所有待处理的迁移
faasjs-pg up 运行下一个待处理的迁移
faasjs-pg down 回滚最近一次应用的迁移

要求：

  • 对于 status、migrate、up 和 down，必须设置 DATABASE_URL 环境变量。
  • 迁移文件默认位于 ./migrations 中。
  • 迁移文件命名约定：<timestamp>-<name>.ts（由 faasjs-pg new 自动生成）。
  • 参见 PG Schema 和迁移指南了解迁移编写规则。

示例：

DATABASE_URL=postgres://localhost:5432/myapp npx faasjs-pg migrate
DATABASE_URL=postgres://localhost:5432/myapp npx faasjs-pg new add_users_table

类型生成工作流

保持类型声明与 API 路由同步：

  1. 创建、重命名或移动 .api.ts 文件后，运行：

    npx faas types

  2. 生成的文件写入到 src/.faasjs/types.d.ts。

  3. 输出通过模块增强在 @faasjs/types 上声明 FaasActions 接口，为每个路由提供类型化的 InferFaasAction 和 InferFaasApi。

  4. faas types 是幂等的：它将生成的内容与现有文件进行比较，仅在内容有变化时才写入。

  5. 要从代码中生成类型（例如在脚本中），从 @faasjs/dev 导入 generateFaasTypes：

    import { generateFaasTypes } from '@faasjs/dev'

const result = await generateFaasTypes({ root: process.cwd() })
console.log(result.output, result.routeCount)

测试命令

命令 描述
vp test 运行完整的测试套件
vp test <pattern> 运行匹配文件名校验的测试（例如 vp test list 运行 *list*）
vp test --watch 文件变更时重新运行测试
FaasLog=info vp test 使用特定日志详细程度运行测试

参见测试指南了解测试原则，以及项目配置指南了解 Vitest 项目配置。

常见命令错误及恢复

faas: command not found

  • 检查：npx faas --version 或 npx @faasjs/dev --version
  • 恢复：确保已安装 @faasjs/dev：npm install @faasjs/dev
  • 根本原因：faas 二进制在 @faasjs/dev 中定义，不是全局命令。

vp: command not found

  • 检查：npx vp --version
  • 恢复：确保已安装 vite-plus：npm install vite-plus
  • 根本原因：vp 二进制由 vite-plus 提供。

faas types 失败

  • 检查：src/faas.yaml 是否存在？YAML 是否有效？
  • 检查：src/ 是否至少包含一个 .api.ts 文件？
  • 恢复：如果在项目根目录外，运行 npx faas types --root .。
  • 参见：faas.yaml 规范

faasjs-pg 命令失败

  • 检查：是否设置了 DATABASE_URL？echo $DATABASE_URL
  • 检查：数据库是否可达？psql $DATABASE_URL -c 'SELECT 1'
  • 恢复：在命令前添加 DATABASE_URL=postgres://... 或将其设置在 .env 中。
  • 根本原因：除 new 之外的所有 faasjs-pg 命令都需要活动的数据库连接。

测试失败

  • 检查：依赖是否已安装？npm ls @faasjs/dev vite-plus vitest
  • 检查：项目是否有有效的 vitest.config.ts 或 vite.config.ts？
  • 检查：集成测试所需的环境变量（如 DATABASE_URL）是否可用？
  • 恢复：先运行聚焦测试：vp test src/specific/test.ts
  • 参见：项目配置指南了解 Vite/Vitest 配置。

环境变量和配置

FaasJS 运行时

变量 描述 默认值
FaasEnv faas.yaml 配置加载使用的活动环境名称 development
FaasLog 最低日志级别（debug、info、warn、error） info
FaasLogMode 日志输出风格（plain、pretty） 自动检测
FaasLogSize 长非错误日志的截断阈值（字节） 取决于平台
FaasLogTransport 启用或禁用共享日志传输转发 true

数据库

变量 描述 必需者
DATABASE_URL PostgreSQL 连接字符串 faasjs-pg CLI、@faasjs/pg 客户端引导、@faasjs/pg-dev 测试插件

使用示例

# 使用调试日志运行测试
FaasLog=debug vp test

# 针对特定数据库运行迁移
DATABASE_URL=postgres://user:pass@localhost:5432/myapp npx faasjs-pg migrate

# 为特定项目根目录运行类型生成
npx faas types --root /path/to/project

关键配置文件

文件 用途
faas.yaml 运行时配置：服务器根目录、基础路径、环境覆盖、插件
tsconfig.json TypeScript 配置，继承 @faasjs/types/tsconfig/* 预设
vite.config.ts Vite/Vitest 配置，使用 @faasjs/dev 的 viteConfig
.env 本地开发的环境变量覆盖

规则

  1. 每次 .api.ts 变更后运行 faas types。 创建、重命名或移动 API 文件而不重新生成类型会导致调用方出现类型错误。
  2. 优先使用 vp check --fix 而不是手动格式化。 它通过 vite-plus 共享配置使用 oxlint 和 oxfmt。
  3. 优先使用 vp test 而不是直接调用 vitest。 它使用项目的 vite.config.ts，其中包含所有必要的插件和别名。
  4. 当二进制未全局安装时使用 npx。 faas 和 vp 都是项目 node_modules 的本地命令。
  5. 在运行 faasjs-pg 命令前设置 DATABASE_URL。 除 new 之外的所有操作都需要活动连接。
  6. 保持迁移时间戳唯一。 faasjs-pg new 命令使用 ISO 时间戳；不要手动编辑文件名以避免排序冲突。
  7. 不要手动编辑 src/.faasjs/types.d.ts。 它由 faas types 重新生成，任何手动更改都将被覆盖。
  8. 使用 FaasLog 详细程度进行调试，而不是更改日志调用点。 日志记录器在运行时尊重 FaasLog。

审查清单

  • .api.ts 更改后运行了 faas types（或记录了原因）
  • faas.yaml 是有效的 YAML 并遵循 faas.yaml 规范
  • 提交前通过 vp check --fix
  • vp test 通过（或记录了阻塞因素 + 已运行的更窄验证）
  • 在运行 faasjs-pg 迁移命令前设置了 DATABASE_URL
  • 迁移文件在 migrations/ 中并遵循时间戳命名
  • 未手动编辑 src/.faasjs/types.d.ts
  • 当二进制未全局安装时使用了 npx 前缀
  • 环境变量（FaasEnv、FaasLog、DATABASE_URL）在每个项目中都有文档或显而易见
